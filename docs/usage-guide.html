体验场景识别与还原

exp-scene-recognition

从碎片化的用户反馈、服务录音、投诉记录或行为数据中，识别用户处于什么场景、触发条件是什么、情绪状态如何，生成结构化场景卡。

触发：收到用户反馈 / 录音 / 投诉 / 行为数据 输出：场景卡（结构化）

场景卡包含

  • 场景类型 · 体验对象 · 触发条件
  • 用户目标 · 当前阻力 · 情绪状态
  • 业务影响 · 体验机会 · 指标观察 · 可复用规则

如何使用这个 Skill

单点说明如何触发
直接对 Agent 说：帮我识别这个用户反馈的场景，或上传一段客服录音/投诉文本后说 用场景识别还原一下。
结果说明你会得到什么
一张结构化的场景卡，可以直接用于后续 JTBD 分析或旅程分析，也可以作为资产沉淀到知识库。
补充 Q&A常见问题
场景识别和旅程分析有什么区别？
场景识别聚焦"用户在哪个具体情境下遇到问题"，是单点还原；旅程分析覆盖"用户从进入到离开的完整过程"，是全景还原。建议先用场景识别定位问题点，再用旅程分析展开全貌。
情绪状态是怎么判断的？
基于用户原声中的情绪词、标点符号、句式强度、以及行为数据（如反复操作、短停留）综合判断，输出为"平静 / 焦虑 / 愤怒 / 失望 / 惊喜"五档。

JTBD 用户任务提炼

exp-jtbd-extraction

从用户行为、表达和选择中提炼功能任务、情绪任务和社会任务，识别用户真正想完成的"进步"是什么，以及当前的替代方案和阻力。

触发：用户动机不明确 / 行为与预期不一致 输出：JTBD 卡片

JTBD 卡片结构

  • 功能任务：帮我……（具体要完成什么事）
  • 情绪任务：让我感到……（使用过程中的情绪期待）
  • 社会任务：让我看起来……（在他人眼中的形象期待）
  • 当前替代方案 · 当前阻力 · 体验机会

如何使用这个 Skill

单点说明如何触发
说：帮我提炼这个用户的 JTBD，或 用户说"不知道怎么用"背后的任务是什么？
结果说明你会得到什么
一张 JTBD 卡片，拆解用户的三个任务维度。可直接用于指导功能设计优先级和营销文案方向。
补充 Q&A常见问题
JTBD 和普通需求分析有什么不同？
普通需求分析关注"用户说要什么"，JTBD 关注"用户想通过产品完成什么进步"。JTBD 更接近用户的真实动机，能揭示用户为什么选择（或放弃）你的产品。
"社会任务"听起来很抽象，能举个例子吗？
比如用户用 Notion 做笔记，功能任务是"记录信息"，社会任务是"让我在同事眼中看起来专业、有条理"。社会任务直接影响用户的分享和传播行为。
🧠

用户认知心理学分析

exp-cognitive-analysis

从认知负荷、心智模型、预期管理、控制感、损失厌恶、信任形成、记忆偏差七个维度检查体验设计，判断用户是否看得懂、找得到、记得住、信得过、感到可控。

触发：理解困难 / 决策犹豫 / 信任不足 / 记忆偏差 输出：认知心理检查清单 + 策略建议

七维度检查项

  • 认知负荷：信息是否超出工作记忆容量（7±2 原则）
  • 心智模型：是否符合用户已有心理模型
  • 预期管理：系统状态是否及时反馈
  • 控制感：是否提供撤销 / 返回 / 取消
  • 损失厌恶：是否避免不必要的决策阻碍
  • 信任形成：信任信号（评价、认证、保障）是否充足
  • 记忆偏差：是否考虑峰终定律（峰值 + 结尾决定整体评价）

如何使用这个 Skill

单点说明如何触发
说：用认知心理学分析一下这个注册流程，或 为什么用户在这里大量流失？从认知角度分析
结果说明你会得到什么
一份七维度检查清单，每个维度标注"通过 / 有风险 / 严重问题"三级，并给出对应的优化建议。可直接作为设计评审的检查项。
补充 Q&A常见问题
这个分析和启发式评估（Skill 21）有什么区别？
启发式评估关注界面和交互是否符合通用可用性原则；认知心理学分析关注用户的内心处理过程——用户是否能理解、记住、信任这个设计。两者互补，建议一起使用。
"峰终定律"具体怎么用？
用户在回忆体验时，只记得"峰值时刻"（最强烈的正向或负向情绪）和"结尾时刻"。设计时应刻意打造一个正向峰值（如"完成"时的成就感），并确保结尾是正面的（如感谢页、优惠券）。

VOC 洞察分析

exp-voc-insight

对批量客户原声（VOC）进行分类、识别高频问题、评估情绪强度、匹配场景、提炼 JTBD、挖掘深层需求，输出优先级排序和策略机会。

触发：收到批量 VOC / 客服对话 / 投诉 / App 评价 输出：VOC 洞察分析报告

报告结构

  • 数据范围 · 样本量 · 数据源说明
  • 高频问题分类 · 每类代表性原声（直接引用）
  • 情绪强度分布（正向 / 中性 / 负向占比）
  • 对应场景匹配 · JTBD 提炼 · 深层需求挖掘
  • 优先级矩阵（影响人数 × 情绪强度）× 策略机会

如何使用这个 Skill

单点说明如何触发
上传 VOC 数据文件（Excel / CSV）后说：帮我分析这些用户评价的洞察，或直接粘贴一段客服对话记录。
结果说明你会得到什么
一份VOC 洞察分析报告，包含问题分类、情绪分布、JTBD 提炼和优先级排序。可直接用于指导产品迭代优先级会议。
补充 Q&A常见问题
VOC 分析需要多少条数据才够？
定性洞察 30-50 条已能发现主要模式；定量统计建议 200 条以上。如果数据量较少，Agent 会标注"样本量不足，结论仅供参考"。
"深层需求挖掘"是怎么做的？
在 JTBD 基础上，进一步追问"用户为什么在意这个？"——比如用户抱怨"找不到按钮"的深层需求可能是"我感到失控和焦虑"，对应的体验机会是"增强可预测性和控制感"。

用户画像构建 NEW

exp-persona-building

基于调研数据、行为数据和业务数据，构建结构化的用户画像，包括人口属性、行为特征、需求动机、痛点和体验偏好，指导产品设计和体验决策。

触发：需要构建用户画像 / 指导产品设计 输出：用户画像卡（1 或多个）

画像维度

  • 基本信息：角色定位 · 人口属性 · 使用场景
  • 行为特征：使用频率 · 核心功能 · 设备偏好
  • 需求动机：核心 JTBD · 使用目标 · 决策因素
  • 痛点与期望：当前阻力 · 替代方案 · 期望体验
  • 体验敏感度：对哪些体验维度最敏感（速度 / 美观 / 稳定…）

如何使用这个 Skill

单点说明如何触发
说：帮我构建一个 25-35 岁自由职业者的用户画像，或提供调研数据后说 基于这些数据生成用户画像。
结果说明你会得到什么
1 个或多个用户画像卡（含名字、照片描述、画像维度完整填写）。可直接贴在 PRD、设计评审文档或看板上供团队参考。
补充 Q&A常见问题
用户画像和同理心地图（Skill 18）有什么区别？
用户画像是"这个用户是谁"——客观描述；同理心地图是"这个用户内心在想什么"——主观共情。建议先构建画像（明确研究对象），再用同理心地图深入理解其内心世界。
画像需要多详细？会不会过度细化？
画像的目的是"让团队对用户产生共情"，不是写简历。建议每个产品聚焦 3-5 个核心画像，每个画像一页 A4 足够。避免过度细化导致"假精确"（比如精确到"喜欢蓝色第 3 朵向日葵"这种无用细节）。

同理心地图 NEW

exp-empathy-map

深入理解用户在某场景下的所思、所想、所做、所感，从"说 / 想 / 做 / 感"四个象限还原用户的内心世界，建立团队对用户的共情认知。

触发：需要深入理解用户内心 / 建立团队共情 输出：同理心地图（四象限）

四象限结构

  • Says（说）：用户直接表达的话语、反馈、评价
  • Thinks（想）：用户未说出口的想法、顾虑、期待
  • Does（做）：用户的实际行动、操作、替代行为
  • Feels（感）：用户的情绪波动、满意度、焦虑来源
  • 综合产出：Pains（痛点）· Gains（收益期待）

如何使用这个 Skill

单点说明如何触发
说：帮我画一个"深夜加班用 App 点外卖"的同理心地图，或在用户画像基础上说 给这个画像做同理心地图。
结果说明你会得到什么
一张四象限同理心地图，可直接用于团队共情工作坊、设计讨论前的"带入用户视角"环节。输出格式支持 Markdown 表格或可视化四格图描述。
补充 Q&A常见问题
"想"和"感"看起来很相似，怎么区分？
"想"是认知层面（用户在琢磨什么、担心什么）；"感"是情绪层面（用户是焦虑、兴奋还是沮丧）。比如："想"= "这个流程会不会很复杂？"；"感"= "感到焦虑，因为怕做错"。
同理心地图做完之后怎么用？
在 Pains 和 Gains 的基础上，直接转化为设计机会点："用户担心复杂 → 提供进度提示和预估完成时间"。建议将同理心地图贴在团队工作区，设计讨论前先"带入用户视角"。

旅程还原与分析

exp-journey-analysis

先定义旅程尺度（微型 / 小型 / 中型 / 大型 / 生命周期）和边界，基于证据还原实际旅程，再设计理想旅程，最后输出差距分析、关键断点和改善机会。

触发：需要理解完整体验过程 / 诊断断点 输出：旅程分析报告（实际 vs 理想）

五种旅程尺度

  • 微型旅程：单次操作（几秒～几分钟）—— 如"修改头像"
  • 小型旅程：单次使用会话（几分钟～几十分钟）—— 如"完成一次下单"
  • 中型旅程：跨天或跨周的多步流程 —— 如"完成一次报销申请"
  • 大型旅程：跨月或跨季度的持续使用 —— 如"使用产品完成一个项目"
  • 生命周期旅程：从认识到离开的完整过程 —— 如"用户全生命周期"

如何使用这个 Skill

单点说明如何触发
说：帮我还原用户从安装到首次使用的旅程，或 分析一下用户退订服务的完整旅程，找出所有断点。
结果说明你会得到什么
一份旅程分析报告，包含：实际旅程阶段表（每阶段用户行为、情绪、痛点）+ 理想旅程阶段表 + 差距分析矩阵 + 关键断点排序 + 改善机会优先级。可直接用于产品路线图规划。
补充 Q&A常见问题
"实际旅程"和"理想旅程"的数据从哪来？
实际旅程基于真实证据（埋点数据、用户访谈、客服记录）；理想旅程基于用户目标和 JTBD（用户希望怎样完成）。两者的差距就是体验机会点。
旅程尺度选错了会怎样？
尺度太小会遗漏关键上下文（比如只分析"支付"而忽略"为什么用户来到支付"）；尺度太大会让分析过于宏观、难以落地。建议从小型旅程开始，根据需要放大或缩小。

峰值体验设计

exp-peak-design

识别旅程中最影响用户记忆和评价的关键时刻，设计安心峰值、效率峰值、被理解峰值、专业峰值、惊喜峰值或结尾峰值。

触发：旅程分析发现体验低谷 / 需要强化品牌记忆 输出：峰值体验设计卡

六类峰值设计

  • 安心峰值：让用户感到安全、放心、被保护（如支付安全提示）
  • 效率峰值：让用户感到顺畅、快速、省力（如"3 步完成"）
  • 被理解峰值：让用户感到被理解、被重视（如个性化推荐）
  • 专业峰值：让用户感到专业、可靠、有品质（如数据可视化报告）
  • 惊喜峰值：让用户感到意外、愉悦、超出预期（如意外的优惠券）
  • 结尾峰值：让用户带走正面记忆（如感谢页 + 下一步引导）

如何使用这个 Skill

单点说明如何触发
说：在用户完成订单的旅程节点设计一个峰值体验，或先完成旅程分析后说 基于这个旅程分析结果，设计峰值体验。
结果说明你会得到什么
一张峰值体验设计卡，明确：关键场景、用户状态、当前低谷、峰值类型、具体设计动作、用户感受描述、品牌印记、衡量指标。可直接交给设计团队落地。
补充 Q&A常见问题
是不是每个旅程都需要设计峰值？
不需要，也做不到。峰终定律告诉我们用户只记得峰值和结尾。建议每个旅程最多设计 1 个峰值 + 1 个结尾峰值，聚焦资源打造最关键的记忆点。
"惊喜峰值"会不会让用户觉得被套路？
会，如果惊喜和用户的 JTBD 无关。好的惊喜峰值必须和用户目标相关（比如用户在找酒店时送出"周边美食地图"，而不是无关优惠券）。无关的惊喜容易被视为干扰或套路。

服务蓝图绘制

exp-service-blueprint

将用户行为、前台触点、员工动作、后台流程、支撑系统、证据层同步展开，识别各层的痛点和机会点，完成服务设计检查。

触发：需要分析服务完整交付过程 / 前后台协同问题 输出：服务蓝图（六层矩阵表）

蓝图六层（从上到下）

  • 用户行为层：用户在旅程中每一步做了什么
  • 前台触点层：用户直接接触的界面 / 人员 / 物料
  • 前台员工层：一线员工在每个触点的动作
  • 后台流程层：支持前台的后台流程和规则
  • 支撑系统层：IT 系统、数据、工具支持
  • 证据层：用户离开后留下的实体 / 数字证据（小票、邮件、评价）

如何使用这个 Skill

单点说明如何触发
说：帮我画一下"用户到店用餐"的服务蓝图，或提供旅程分析结果后说 把这个旅程展开成服务蓝图。
结果说明你会得到什么
一份六层服务蓝图（表格形式），横向是旅程阶段，纵向是六层。每个格子标注该层的动作、痛点或机会。可直接用于服务设计评审和跨部门对齐。
补充 Q&A常见问题
服务蓝图和旅程地图（Skill 5）有什么区别？
旅程地图只关注用户侧（用户做了什么、感受如何）；服务蓝图同时展开用户侧 + 企业侧（前台员工、后台流程、支撑系统）。旅程地图是服务蓝图的第一层。
"证据层"是什么意思？为什么重要？
证据层是用户带走的东西（小票、包裹、邮件、记忆），直接影响用户的事后传播和再次使用。比如：外卖小票上印一句暖心的话，就是增加正面证据，促使用户拍照分享。
👤

员工体验诊断

exp-employee-diagnosis

分析员工 JTBD、服务目标与用户期待的匹配度，检查信息支持、工具支持、知识支持、权限支持、情绪压力和协作成本，评估员工体验如何传导到客户体验。

触发：客户体验问题可能源于员工侧 输出：员工体验分析报告

诊断维度

  • 员工 JTBD：功能任务（帮我完成服务）· 情绪任务（让我不焦虑）· 社会任务（让我看起来专业）
  • 四项支持度：信息 · 工具 · 知识 · 权限
  • 情绪压力：情绪劳动负荷 · 负面情绪传递风险
  • 传导机制：员工体验 → 服务态度 → 客户体验的传导路径

如何使用这个 Skill

单点说明如何触发
说：客服响应慢，帮我诊断是不是员工体验的问题，或 分析一下员工体验对客户满意度的影响。
结果说明你会得到什么
一份员工体验诊断报告，明确指出：哪些员工侧问题正在影响客户体验、差距在哪里、改善优先级是什么。可直接用于 HR 和运营团队的改进计划。
补充 Q&A常见问题
"员工体验传导"有理论依据吗？
有。服务利润链理论（Heskett et al.）明确指出：员工满意度 → 员工忠诚度 → 服务品质 → 客户满意度 → 客户忠诚度 → 盈利能力。这个 skill 帮助你找到传导链中的断裂点。
这个分析和 NPS 调查有什么区别？
NPS 调查看结果（用户是否推荐）；员工体验诊断看原因（为什么服务不好）。两者结合使用：NPS 发现"客服满意度低"，再用员工体验诊断找出"是工具不好用还是权限不够"。

场景化体验营销设计

exp-scenario-marketing

基于用户场景中的任务、情绪和决策阻力，选择最合适的营销类型，设计触达时机、内容、体验动作和转化路径。

触发：需要在特定场景触达用户 / 推动转化 / 增强留存 输出：场景化体验营销分析卡

五类营销类型

  • 认知型：唤醒需求、建立认知（用户还不知道自己有这个问题）
  • 决策型：降低决策阻力、推动转化（用户知道问题，在比较方案）
  • 使用型：提升体验、增强粘性（用户已购买，需要用好）
  • 关系型：深化关系、提升忠诚度（用户是回头客）
  • 挽回型：重新激活、挽回流失（用户已流失或即将流失）

如何使用这个 Skill

单点说明如何触发
说：用户在注册后 7 天没登录，设计一个挽回营销方案，或结合场景识别结果说 在这个场景下设计一个决策型营销。
结果说明你会得到什么
一张场景化营销分析卡，明确：目标场景、用户 JTBD、情绪状态、决策阻力、触达时机、触达内容、体验动作、转化路径、品牌印记、衡量指标。可直接交给增长团队执行。
补充 Q&A常见问题
"体验营销"和普通营销有什么区别？
普通营销关注"让用户买"；体验营销关注"让用户在过程中感到好"。体验营销的每一步都是一个体验机会，而不仅仅是转化机会。比如：决策型营销不只是"限时优惠"，而是"帮你做决定的工具 + 优惠"。
触达时机怎么判断？太早或太晚会怎样？
最佳触达时机是用户 JTBD 被激活但未完成的时刻。太早（用户还没意识到问题）→ 被打扰；太晚（用户已放弃或已用别的产品解决了）→ 无意义。Agent 会基于旅程分析结果推荐最佳时机。

体验编排设计

exp-orchestration-design

围绕"触发条件 → 判断规则 → 响应动作 → 人工介入 → 指标监测 → 资产回写"结构，设计 Agent 自动化体验流程，明确什么情况下自动响应、什么情况下转人工。

触发：需要将旅程节点转化为 Agent 自动化机制 输出：旅程编排卡（可转为开发需求）

编排六步结构

  • 触发条件：行为触发 · 数据触发 · 时间触发 · 情绪触发
  • 判断规则：用户分层判断 · 场景匹配度 · 优先级计算 · 渠道选择
  • 响应动作：推送内容 · 提供帮助 · 执行操作（如发券）· 记录数据
  • 人工介入条件：用户明确要求 · 问题无法解决 · 高风险操作 · 情绪升级
  • 指标监测：转化率 · 满意度 · 人工介入率 · 持续优化方向

如何使用这个 Skill

单点说明如何触发
说：设计一个"用户注册后 3 天未登录"的自动挽留编排，或 把"用户投诉"这个场景做成自动识别和响应的编排。
结果说明你会得到什么
一张旅程编排卡，结构化的六步设计，可直接交给开发团队转化为代码逻辑。也支持导出为决策表或流程图格式。
补充 Q&A常见问题
"人工介入"的边界怎么定？太早转人工浪费成本，太晚影响体验。
三个信号必须转人工：（1）用户明确说"转人工"；（2）Agent 置信度低于阈值（如 <70%）；（3）涉及高风险操作（如退款、改地址）。建议在编排卡中明确写出这三条规则。
这个 skill 的输出能直接开发吗？还需要什么？
编排卡提供了逻辑结构，但开发前还需要：（1）确定数据来源（触发条件需要哪些埋点）；（2）设计具体的话术/推送内容；（3）定义异常兜底逻辑。Agent 会在输出时提示这些待确认项。

体验指标体系设计

exp-metric-design

从体验感知、行为、服务效率、质量、员工体验五类指标中选取合适的指标，为每个指标定义计算方式、数据来源、预警阈值、关联痛点和责任方。

触发：需要建立体验测量体系 / 为体验问题建立判断标准 输出：指标卡集合（每个指标一张卡）

五类指标

  • 行为指标：转化率 · 留存率 · 使用频率 · 流失率（用户实际行为）
  • 服务效率指标：首次响应时间 · 平均解决时长 · 一次解决率
  • 质量指标：错误率 · 可用性得分 · 系统稳定性

如何使用这个 Skill

单点说明如何触发

结果说明你会得到什么
一组指标卡（每个指标包含：定义、计算公式、数据来源、预警阈值、关联痛点、责任方、响应动作）。可直接用于数据看板和预警规则配置。
补充 Q&A常见问题
感知指标（问用户）和行为指标（看数据）哪个更可靠？
两者互补。感知指标告诉你用户"怎么说"（可能受礼貌偏误影响）；行为指标告诉你用户"怎么做"（更真实，但不知道原因）。建议每个关键体验都同时测量两类指标。
"预警阈值"怎么定？有没有行业标准？
分三步：（1）行业基准（Agent 会提供）；（2）自身历史数据（计算过去 3 个月均值 ± 1 个标准差）；（3）业务影响关联（这个指标下降 10% 是否对应业务指标下降）。三者结合定阈值。

体验问卷设计 NEW

exp-survey-design

基于体验测量目标，设计结构化的体验问卷，包括指标选择、量表设计、题型设计、问卷结构编排和投放策略。

触发：需要设计体验测量问卷 / 选择量表 / 制定投放策略 输出：问卷设计方案（含题目原文）

常用量表类型

  • CSAT：1-5 分（非常不满意 → 非常满意）
  • NPS：0-10 分（推荐意愿）
  • CES：1-5 或 1-7 分（费力程度，分数越低越轻松）
  • CSATG：5 档表情符号（适合移动端）
  • Likert：5 点或 7 点认同度（"非常不同意"→"非常同意"）

问卷七步结构

  • 开场白（建立信任）→ 筛选题（确保答对人）
  • → 核心量表题（CSAT/NPS/CES）
  • → 维度诊断题（具体哪里好 / 哪里不好）
  • → 行为题（频率、偏好）→ 开放题（为什么给这个分）→ 结束语（感谢 + 激励）

如何使用这个 Skill

单点说明如何触发
说：帮我设计一份测量"客服体验"的问卷，或 我要测 NPS，帮我设计问卷题目。
结果说明你会得到什么
一份完整问卷设计方案，包含：问卷目标、适用量表、完整的题目原文（可直接复制到问卷工具）、逻辑跳转规则、投放时机建议、样本量计算。可直接用于问卷星、Typeform 等工具。
补充 Q&A常见问题
问卷应该多长？用户不愿意填怎么办？
核心量表题 1-3 题（必须），维度诊断题 3-5 题（推荐），开放题 1 题（可选）。总计 不超过 8 题。提高填写率的方法：在用户情绪峰值（刚完成操作 / 刚联系完客服）时触发，并提供小额激励。
NPS 和 CSAT 应该同时问吗？会不会重复？
不重复，测的东西不同。CSAT 测"这次满意吗"（单次体验）；NPS 测"会推荐吗"（整体忠诚度）。建议同时问：CSAT 用于迭代优化，NPS 用于衡量长期健康度。但注意不要连续问多个满意度题，会导致用户疲劳。

满意度统计与分析 NEW

exp-satisfaction-stats

对 CSAT、NPS、CES 等满意度指标进行统计计算、分组分析、趋势追踪和综合诊断，包含计算公式、基准参考和预警机制。

触发：需要计算 / 分析满意度指标 / 生成统计报告 输出：满意度统计报告

三大指标计算

  • CSAT =（4 分 + 5 分人数 ÷ 总有效样本）× 100%，范围 0-100%
  • NPS = 推荐者（9-10 分）% - 贬损者（0-6 分）%，范围 -100 到 +100
  • CES = 平均分（1-5 或 1-7 分制），也可报告"低费力（4-5 分）占比"

综合诊断框架

  • CSAT 高 + NPS 高 + CES 低 → 体验优秀，保持优势
  • CSAT 高 + NPS 低 + CES 低 → 满意但不忠诚，需强化情感连接
  • CSAT 低 + NPS 低 + CES 高 → 紧急改善核心流程和易用性

如何使用这个 Skill

单点说明如何触发
上传问卷数据文件后说：帮我分析这份 CSAT 调查的统计结果，或 计算 NPS 并分析不同渠道的差异。
结果说明你会得到什么
一份满意度统计报告，包含：各指标得分、分组对比（如不同渠道 / 不同用户群的差异）、趋势图（如有历史数据）、综合诊断结论、预警提示。可直接用于管理层汇报。
补充 Q&A常见问题
NPS 多少分算"好"？有没有行业基准？
行业差异很大。大致参考：+50 以上算优秀（科技 / SaaS 行业）；+30~+50 算良好；0~+30 算及格；负数需紧急改善。Agent 会在报告中附上你所处行业的基准数据。
样本量多少才有统计意义？
NPS 需要约 100 份有效样本才能稳定（标准差约 10 时，±3% 误差需约 1000 份；±5% 误差需约 400 份）。如果样本量小，Agent 会标注置信区间，提醒谨慎解读。

可用性测试 NEW

exp-usability-testing

设计可用性测试方案，编写测试任务脚本，执行测试并分析结果。涵盖形成性测试（早期发现）和总结性测试（量化评估）。

测试设计要素

  • 参与者：招募标准 · 样本量（5-8 人可发现 80% 问题）· 激励
  • 任务脚本：3-5 个典型任务 · 场景设定 · 成功标准
  • 测试指标：任务完成率 · 完成时间 · 错误次数 · 满意度评分 · SUS 量表

如何使用这个 Skill

单点说明如何触发
说：帮我设计一份针对"首次注册流程"的可用性测试方案，或测试完成后说 帮我分析这 5 位用户的测试数据。
结果说明你会得到什么
一份可用性测试报告，包含：测试方案（目标、参与者、任务脚本）+ 测试结果（完成率、时间、错误分析）+ 问题清单（按严重度排序）+ 改进建议优先级。可直接用于设计迭代。
补充 Q&A常见问题
为什么只测 5-8 个人？样本量是不是太少了？
著名可用性专家 Jakob Nielsen 的研究表明：5 个用户能发现约 80% 的可用性问题。第 6-8 个用户主要起到验证作用。如果预算充足，可以测更多，但边际效用递减。关键是早点测、多次测，而不是一次性测很多人。
"形成性测试"和"总结性测试"有什么区别？
形成性测试在设计过程中做，目标是"发现问题、指导改进"（定性为主）；总结性测试在发布前做，目标是"量化评估、判断是否可用"（定量为主，如 SUS 得分）。两者都重要，建议都做。

启发式评估 NEW

exp-heuristic-evaluation

基于 Nielsen 十大可用性启发式原则，系统性评估产品界面和交互流程中的可用性问题，快速识别问题并按严重度分级。

触发：需要快速评估产品可用性问题 / 设计阶段自查 输出：启发式评估报告（含严重度分级）

Nielsen 十大原则

  • 系统状态可见性 · 系统与现实匹配 · 用户控制与自由
  • 一致性与标准化 · 错误预防 · 识别而非回忆
  • 灵活性与效率 · 简约设计 · 帮助用户识别并纠正错误 · 帮助文档

问题严重度分级

  • 0 级：不是问题
  • 1 级：小问题，用户体验有轻微不便
  • 2 级：中等问题，影响用户体验但可绕过
  • 3 级：严重问题，显著阻碍用户完成任务
  • 4 级：灾难性问题，用户完全无法完成任务

如何使用这个 Skill

单点说明如何触发
说：用启发式评估检查一下这个注册流程（并附上截图或链接），或 帮我按 Nielsen 十大原则评估这个设计。
结果说明你会得到什么
一份启发式评估报告，按十大原则分类列出发现的问题，每个问题标注严重度（0-4 级）、影响描述、改进建议。可直接作为设计评审的输入。
补充 Q&A常见问题
启发式评估和可用性测试（Skill 20）先做哪个？
建议先做启发式评估（快速、低成本，发现问题后先修复），再做可用性测试（找真实用户验证）。启发式评估相当于"内部自查"，可用性测试相当于"外部验证"，两者互补。
严重度评级会不会太主观？
会有一定主观性。降低主观性的方法：（1）多人独立评估后讨论共识；（2）结合数据（如该页面流失率确实高，则严重度上调）；（3）以用户目标为锚（是否阻碍用户完成 JTBD）。Agent 会提示置信度。

竞品体验分析 NEW

exp-competitive-analysis

从用户体验视角系统对比竞品，发现体验差距和差异化机会。涵盖功能体验、交互流程、服务触点、品牌体验等多维度结构化对比。

触发：需要对比竞品体验 / 发现差异化机会 输出：竞品体验分析报告

分析维度

  • 功能体验对比：核心功能覆盖度 · 差异化能力 · 便捷性
  • 交互流程对比：操作步数 · 认知负荷 · 异常处理 · 首次体验（FTUE）
  • 服务触点对比：关键时刻体验 · 客服质量 · 售后保障
  • 品牌体验对比：品牌调性 · 情感连接 · 用户口碑

如何使用这个 Skill

单点说明如何触发
说：帮我对比分析 A 产品和 B 产品在注册体验上的差异，或 分析一下竞品的客服体验，找出我们的差距。
结果说明你会得到什么
一份竞品体验分析报告，包含：竞品列表、各维度对比矩阵、体验差距雷达图（文字描述）、差异化机会排序、对自身产品的建议。可直接用于产品规划和差异化定位。
补充 Q&A常见问题
应该选几个竞品分析？选直接竞品还是间接竞品？
建议选 2-3 个直接竞品（同品类、同目标用户）+ 1 个跨界标杆（体验特别好的非竞品，如服务体验可参考海底捞，App 体验可参考抖音）。跨界标杆能带来突破性的体验灵感。
"体验差距"怎么量化？不能只靠感觉吧？
Agent 会使用体验评分矩阵：每个维度按 1-5 分评分，并附上具体证据（如"竞品注册只需 2 步，我们需要 5 步"）。关键是有具体证据支撑，而不是"感觉竞品更好"。
🔎

触点审计 NEW

exp-touchpoint-audit

全面盘点用户与产品 / 品牌的所有接触触点，评估各触点体验质量，发现覆盖盲区，输出触点健康度和优化优先级。

触发：需要盘点所有触点 / 评估触点质量 / 发现覆盖盲区 输出：触点审计报告

审计框架

  • 触点盘点：按旅程阶段枚举所有触点，标注触点类型（人工 / 数字 / 物理）
  • 触点质量评估：可用性 · 一致性 · 情感体验 · 信息准确度 · 响应速度
  • 盲区识别：用户需要但缺失的触点 · 被竞品覆盖但自身缺失的触点
  • 触点健康度矩阵：影响力（横轴）× 当前质量（纵轴）→ 四象限优先级

如何使用这个 Skill

单点说明如何触发
说：帮我盘点用户与我们的所有接触触点，或 审计一下我们在"售后"阶段有哪些触点，质量如何。
结果说明你会得到什么
一份触点审计报告，包含：完整触点清单（表格）、每个触点的质量评分、盲区清单、触点健康度矩阵（优先级四象限）、改善路径建议。可直接用于体验全景图绘制。
补充 Q&A常见问题
"触点"和"旅程阶段"是什么关系？
旅程阶段是时间轴（用户先做什么、后做什么）；触点是接触点（在每个阶段用户和哪些人 / 界面 / 物料接触）。一个旅程阶段可能包含多个触点，一个触点也可能跨越多个阶段。
发现盲区后，是不是所有盲区都要补上？
不是。先评估盲区的影响力：高频场景中的盲区 > 低频场景中的盲区；情感高风险场景中的盲区 > 功能性场景中的盲区。用触点健康度矩阵（影响力 × 质量）来排优先级，集中力量补最有价值的盲区。
能力域 06 · 分群与运营（1个）
🎯

用户分群与分层 NEW

exp-user-segmentation

基于体验需求对用户进行分群，设计差异化体验策略，建立用户分层运营体系。

触发：需要设计差异化体验策略 / 建立分层运营体系 输出：用户分群方案 + 差异化策略地图

分群维度

  • 价值分层：RFM 模型（最近 / 频率 / 金额）· LTV 分层 · 流失风险分层
  • 行为分群：使用频率 · 功能偏好 · 行为路径聚类
  • 体验敏感度分群：高敏感（在意细节）· 中敏感 · 低敏感（只要能用就行）
  • 场景分群：按使用场景和目标聚类（如"深夜加班族""周末休闲族"）

如何使用这个 Skill

单点说明如何触发
说：帮我按体验敏感度给用户分群，并设计差异化策略，或提供用户数据后说 基于这些数据做 RFM 分群。
结果说明你会得到什么
一份用户分群方案（含每群的用户特征描述）+ 一张差异化策略地图（每群对应的体验优先级、资源配置、触达策略）。可直接用于增长团队的分层运营计划。
补充 Q&A常见问题
分群越多越好吗？会不会导致运营太分散？
不是。建议聚焦 3-5 个核心分群，每个分群都要有足够的用户量和明确的差异化需求，才值得单独制定策略。分群太多会导致资源分散，每个群都做得不深。
"体验敏感度分群"是怎么判断的？
通过三个信号判断：（1）行为信号——高敏感用户会更频繁地反馈问题、更仔细地阅读说明；（2）VOC 信号——高敏感用户的抱怨更细致；（3）付费意愿——高敏感用户更愿意为体验付费。Agent 会综合这些信号给出分群建议。
能力域 07 · 商业价值（1个）
💰

体验ROI分析 NEW

exp-roi-analysis

评估体验投入的商业回报，量化体验对业务指标的影响，为向管理层汇报体验价值提供数据支撑。

触发：需要评估体验投入的商业回报 / 量化体验价值 输出：体验 ROI 分析报告

分析框架

  • 投入量化：人力成本 · 研发成本 · 运营成本 · 机会成本
  • 收益量化：转化提升（金额）· 留存提升（LTV 增加）· 客诉下降（成本节约）· NPS 增长（口碑传播价值）
  • ROI 计算：ROI =（收益 - 投入）/ 投入 × 100%；也可计算 NPV（净现值）、IRR（内部收益率）、回本期
  • 敏感性分析：如果收益只有预期的 50%，ROI 是否仍为正？

如何使用这个 Skill

单点说明如何触发
说：我们投入了 50 万优化注册流程，帮我分析 ROI，或 帮我做一个体验改善项目的商业价值分析，用于向管理层汇报。
结果说明你会得到什么
一份体验 ROI 分析报告，包含：投入明细表、收益计算表、ROI / NPV / IRR 计算结果、敏感性分析（情景模拟）、一页纸管理层汇报摘要。可直接用于预算审批和项目复盘。
补充 Q&A常见问题
体验的收益很难量化，怎么算？（比如"品牌印象提升"怎么算钱？）
用间接量化：品牌印象提升 → NPS 提升 → 口碑推荐带来的新用户获取成本节约 → 算出金额。每个体验指标都可以找到对应的业务指标链条。Agent 会帮你梳理这个链条并给出计算方式。
如果 ROI 算出来是负数，是不是说明体验投入不值得？
不一定。有些体验投入是"防守型"的（如客服体验），ROI 可能为负，但如果不投入，流失会更大（隐含损失远大于投入）。这类投入需要用"不投入的代价"来评估，而不是单纯的 ROI。
能力域 08 · 复盘与沉淀（2个）
🔄

服务体验复盘

exp-service-review

还原服务路径和关键触点表现，识别体验低谷和峰值机会，检查员工体验问题和后台流程问题，评估品牌印记表现，输出改善策略和预警机制。

触发：服务事件结束后 / 需要系统复盘 输出：服务体验复盘报告

复盘框架

  • 事件概述 → 旅程尺度与边界定义 → 用户目标确认
  • 服务路径还原（实际发生了什么）→ 关键触点表现评估
  • 体验低谷识别 → 峰值机会发现 → 员工体验问题检查
  • 后台流程问题诊断 → 品牌印记表现评估
  • 改善策略生成 → 指标与预警机制设计 → 体验资产回写

如何使用这个 Skill

单点说明如何触发
说：上次大促期间客服系统崩溃了，帮我做一次服务体验复盘，或 对新推出的"一键退款"功能做一次体验复盘。
结果说明你会得到什么
一份服务体验复盘报告，结构化的复盘框架输出，明确：发生了什么、为什么发生、哪些做得好（峰值）、哪些做得差（低谷）、如何改进、如何预警。可直接用于团队复盘会议。
补充 Q&A常见问题
复盘和旅程分析（Skill 5）有什么区别？
旅程分析是事前/事中工具（理解用户体验过程，指导设计）；复盘是事后工具（回顾已发生的服务事件，提炼经验教训）。两者都用旅程视角，但目的不同。
复盘报告写出来之后，怎么确保改善动作真的落地？
关键是在复盘报告中明确三件事：（1）每个改善动作的责任人；（2）验收标准（做到什么程度算完成）；（3）跟进时间节点。Agent 会在复盘报告中自动生成这三项的模板，建议直接填入后跟踪。
💾

体验资产沉淀

exp-asset-accumulation

判断产出是否具备复用价值，选择资产类型，填写资产回写模板，明确适用场景、使用方法、边界风险和关联指标。

触发：体验分析 / 服务复盘产生了可复用的产出 输出：资产回写卡（可录入体验资产库）

九种资产类型

  • 场景：已验证的场景卡，可复用于其他产品/功能
  • 旅程模板：某类旅程的标准阶段划分，可套用
  • 指标：验证有效的指标定义和计算方式
  • 话术：验证有效的用户沟通话术
  • 规则：体验设计规则（如"注册流程不超过 3 步"）
  • 案例：典型体验案例（正面 / 负面），用于培训
  • 模板：可复用的分析模板、报告模板
  • 知识条目：体验领域的知识点、原则、最佳实践证明
  • 编排卡：验证有效的 Agent 编排逻辑，可复用于类似场景

如何使用这个 Skill

单点说明如何触发
说：这次分析得出的"注册流程 3 步原则"值得沉淀为体验资产，或任何一个分析/复盘完成后 Agent 会主动问：是否将本次产出沉淀为体验资产？
结果说明你会得到什么
一张资产回写卡，包含：资产名称、类型、来源（来自哪次分析）、适用场景、使用方法、注意事项（边界和风险）、关联指标、更新时间。可直接录入体验资产库供团队复用。
补充 Q&A常见问题
什么样的产出才"值得"沉淀为资产？有没有判断标准？
三条判断标准：（1）可复用性——是否适用于多个场景/项目？（2）已验证性——是否已在真实项目中验证有效？（3）可传承性——新人能否直接拿来用，不需要太多背景解释？三条都满足才值得沉淀。
资产沉淀到哪？是文件还是系统？
Agent 输出的资产回写卡是结构化数据（JSON 格式或 Markdown 表格），可以：（1）存入团队知识库（如 Notion / 腾讯文档）；（2）录入体验资产管理系统的 API；（3）作为 Skill 的补充知识条目。Agent 会根据你的工具环境推荐最合适的沉淀方式。

组合工作流 · 如何使用

单个 Skill 解决单一问题，组合 Skill 解决复杂体验管理问题。以下是 6 条典型工作流及使用说明。

工作流 1：体验诊断全流程

从原始用户反馈到完整策略输出的端到端诊断流程，适用于需要系统诊断一组体验问题并输出改进行动的场景。

① VOC 洞察分析 → ② 场景识别与还原 → ③ JTBD 任务提炼 → ④ 认知心理学分析 → ⑤ 旅程还原与分析 → ⑥ 体验策略生成

如何使用

从 ① VOC 洞察分析开始，将用户反馈数据输入，得到高频问题和 JTBD 初步洞察；接着用 ② 场景识别还原具体场景；再用 ③ JTBD 深化动机理解；然后用 ④ 认知分析检查设计问题；再用 ⑤ 旅程分析展开全貌；最后用 ⑥ 策略生成输出完整改进行动。每个步骤的输出会作为下一步的输入，Agent 会自动串联。

💡 适用场景：收到大量用户投诉后需要系统诊断根本原因并输出改进行动；新产品上线后首次全面体验体检。

工作流 2：体验测量闭环

从指标设计到数据收集再到统计分析的完整测量闭环，适用于需要建立体验测量体系并持续追踪的场景。

① 体验指标体系设计 → ② 体验问卷设计 → ③ 满意度统计分析 → ④ 体验资产沉淀

如何使用

先通过 ① 指标设计确定要测量什么、怎么算；再用 ② 问卷设计将指标转化为可执行的测量工具；收集数据后通过 ③ 满意度统计分析结果与趋势；最后将验证有效的指标定义和问卷模板通过 ④ 资产沉淀存入体验资产库，供后续项目复用。

💡 适用场景：首次建立体验测量体系；季度/年度体验健康度追踪；优化现有问卷和指标体系。

工作流 3：服务优化闭环

从服务分析到优化落地再到持续改进的完整闭环，适用于涉及前后台协同的服务体验优化场景。

① 服务蓝图绘制 → ② 员工体验诊断 → ③ 峰值体验设计 → ④ 体验编排设计 → ⑤ 服务体验复盘

如何使用

用 ① 服务蓝图展开服务的全链路（用户侧 + 企业侧）；通过 ② 员工诊断发现员工侧问题；用 ③ 峰值设计在关键触点打造记忆点；用 ④ 体验编排将优化方案转化为自动化流程；上线后用 ⑤ 服务复盘检验效果并持续迭代。这是一个"分析 → 设计 → 落地 → 复盘"的完整闭环。

💡 适用场景：客服体验优化；线下服务流程重构；跨部门服务协同优化。

工作流 4：增长与留存

从场景洞察到营销干预再到自动化履约的增长流程，适用于需要通过体验干预推动用户增长和留存的场景。

① 场景识别与还原 → ② JTBD 任务提炼 → ③ 用户分群与分层 → ④ 场景化营销设计 → ⑤ 体验编排设计

如何使用

从 ① 场景识别理解用户在哪些场景下流失；用 ② JTBD 理解用户为什么流失；用 ③ 分群 区分不同流失原因的用户群；针对不同分群用 ④ 场景化营销 设计差异化挽回方案；最后用 ⑤ 体验编排 将方案自动化执行。这个流程可以直接提升留存率和 LTV。

💡 适用场景：流失用户挽回；新用户激活（激活率提升）；会员/订阅续费优化。

工作流 5：用户共情与创新

从用户理解到方案创新的共情驱动流程，适用于需要设计以用户为中心的差异化创新方案的场景。

① 用户画像构建 → ② 同理心地图 → ③ JTBD 任务提炼 → ④ 认知心理学分析 → ⑤ 峰值体验设计

如何使用

先用 ① 用户画像明确"我们在为谁设计"；再用 ② 同理心地图深入理解用户的内心世界；然后用 ③ JTBD 提炼用户的真实进步目标；接着用 ④ 认知分析确保设计方案符合用户认知规律；最后用 ⑤ 峰值设计打造让用户记住的体验。这个流程产出的是有共情基础的创新方案，而不是功能堆砌。

💡 适用场景：新产品/新功能创新设计；差异化体验定位；设计冲刺（Design Sprint）中的共情阶段。

工作流 6：质量审计与对标

从内部审计到外部对标的全面评估流程，适用于需要全面评估产品体验质量并输出对标结论的场景。

① 启发式评估 → ② 可用性测试 → ③ 触点审计 → ④ 竞品体验分析 → ⑤ 体验ROI分析

如何使用

用 ① 启发式评估做快速自查（低成本、覆盖广）；再用 ② 可用性测试找真实用户验证（高真实度）；用 ③ 触点审计检查体验覆盖率（有没有盲区）；用 ④ 竞品分析明确体验和行业标杆的差距；最后用 ⑤ ROI 分析为体验改善项目争取预算和资源。这个流程的输出是一份完整的体验审计报告 + 改善优先级建议 + 预算申请依据。

💡 适用场景：年度体验审计报告；体验改版前的全面评估；向管理层申请体验改善预算。
